# 文档切分策略详解

## 1. 先统一文档格式，再做切分

项目不会直接对原始 `md`、`txt`、`docx` 文件分别写一套切分逻辑。

它会先把不同格式的文档统一转成：

```text
MarkdownContent
```

然后后面的切分器只处理这一份统一文本。

例如：

1. `md`
   基本原样成为 `MarkdownContent`

2. `txt`
   纯文本基本原样成为 `MarkdownContent`

3. `docx`
   先解析 Word 内容，再转成 `MarkdownContent`

这样做的好处是：

1. 后续切分逻辑不用关心原始文件格式
2. 不同文档统一进入同一套 chunk 流程
3. Markdown 的标题、图片、表格等结构也方便后续识别

---

## 2. 切分不是固定长度硬切，而是“结构优先，长度约束”

很多初学者会以为：

```text
每 512 个字符切一刀
```

但这个项目不是这样。

它的核心思想是：

1. **优先保留文档结构**
   先尽量按标题、段落、章节、换页、句子边界切

2. **再受 chunk_size 约束**
   每个 chunk 尽量不要太大

3. **最后用 overlap 保留边界语义**
   避免一句话刚好被切断

所以它不是单纯的“按长度切”，而是：

```text
先找适合切的位置，再控制每块大小
```

---

## 3. 默认核心配置

项目核心默认值是：

1. `chunk_size = 512`
2. `chunk_overlap = 80`

这里单位是：

```text
Unicode 字符数
```

不是字节数。

### chunk_size

表示最终 chunk 希望控制在大约 512 字符以内。

### chunk_overlap

表示相邻 chunk 之间尽量保留上一块尾部的一部分内容。

例如：

```text
chunk 1：......RAG 会先检索相关内容，再交给大模型
chunk 2：再交给大模型生成答案，这样可以减少幻觉......
```

这样做是为了避免边界处语义断裂。

### separators

分隔符用于把原始文本递归切成更小的 `unit`，后面再由这些 `unit` 合并成最终 chunk。

当前后端源码里，如果知识库没有配置 `separators`，兜底使用：

1. 空行 `\n\n`
2. 换行 `\n`
3. 中文句号 `。`

切分时会按顺序逐级尝试：

1. 先按空行切，尽量保留完整段落
2. 如果某一段仍然太长，再按换行切
3. 如果还太长，再按中文句号切

`CHUNKING.md` 中推荐的更完整分隔符列表是：

1. 空行 `\n\n`
2. 换行 `\n`
3. 中文句号 `。`
4. 中文感叹号 `！`
5. 中文问号 `？`
6. 英文分号 `;`
7. 中文分号 `；`

完整列表可以把文本继续切到更细的句子边界。

---

## 4. 项目有哪些切分策略

项目支持几种策略：

1. `auto`
2. `heading`
3. `heuristic`
4. `legacy`
5. `recursive`

其中：

### 1. `auto`

自动模式。

它会先分析文档结构，再决定走哪种策略。

### 2. `heading`

按 Markdown 标题切分。

适合：

```md
# 一级标题
## 二级标题
### 三级标题
```

这种结构清晰的 Markdown 文档。

### 3. `heuristic`

启发式结构切分。

适合没有 Markdown 标题，但有明显报告结构的文档，例如：

1. 章节号
2. 换页符
3. 分隔线
4. 报告式段落

比较适合 PDF 转换文本、报告类文档。

### 4. `legacy / recursive`

普通递归分隔符切分。

这是最基础、最稳定的兜底策略，也是你最需要重点理解的部分。

---

## 5. auto 是怎么选策略的

`auto` 本身不是一种具体切法。

它更像一个调度器。

它会先分析文档特征，例如：

1. 有没有大量 Markdown 标题
2. 有没有章节号
3. 有没有换页符
4. 有没有明显视觉分隔符
5. 文档整体更像 Markdown 还是报告

然后决定优先尝试哪种策略。

大致可以理解为：

```text
Markdown 标题明显
-> heading

报告结构明显
-> heuristic

都不明显
-> legacy / recursive
```

而且每种策略切完之后，系统还会检查结果是否合理。

如果切得不好，比如：

1. 大文档只切出一个 chunk
2. 切出了太多很小的 chunk
3. chunk 明显过大

那就会回退到下一种策略。

---

## 6. heading 策略怎么切

`heading` 的核心是：

```text
优先按 Markdown 标题边界切
```

例如：

```md
# RAG 系统

RAG 是检索增强生成。

## 文档入库

文档先上传，再解析，再切分。

## 文档检索

用户提问后，先检索相关 chunk。
```

它会优先形成：

1. `# RAG 系统` 下的一块
2. `## 文档入库` 下的一块
3. `## 文档检索` 下的一块

### 但 heading 仍然受 chunk_size 约束

如果某个标题下内容非常长，比如 1200 字，而 `chunk_size = 512`，那它不能整块保留。

这时会：

1. 先按标题确定 section
2. 再在这个 section 内部调用普通递归切分
3. 把它继续切成多个较小 chunk

所以：

```text
heading 决定优先在哪些地方切
chunk_size 决定一块最多希望多大
```

---

## 7. legacy / recursive 是怎么切的

这是最重要的基础逻辑。

它不是一上来就生成 chunk，而是先生成很多临时片段：

```text
unit
```

然后再把多个 unit 合并成最终 chunk。

整体流程是：

1. 先识别不能随便拆开的内容
2. 按 separators 递归拆成 unit
3. 再把 unit 合并成 chunk
4. 合并时控制 chunk_size
5. 相邻 chunk 之间保留 overlap

---

## 8. 什么是 unit

`unit` 是切分过程中的临时小片段。

它不是最终保存到数据库的 chunk。

你可以把它理解成：

```text
unit = 小积木
chunk = 最终成品块
```

例如原文：

```text
第一段内容。

第二段内容。

第三段内容。
```

按空行切之后，可能先得到：

1. `第一段内容。`
2. 空行
3. `第二段内容。`
4. 空行
5. `第三段内容。`

这些就是 unit。

之后系统再把多个 unit 合并：

```text
chunk 1 = 第一段 + 第二段
chunk 2 = 第三段
```

---

## 9. separators 是什么时候用的

`separators` 就是在**切 unit** 时使用的。

它决定的是：

```text
原始文本要先按哪些边界拆成小片段
```

默认递归过程可以理解为：

1. 先按空行 `\n\n` 切，把文本拆成段落级 unit
2. 如果某个段落 unit 仍然超过 `chunk_size`，再在这个 unit 内按换行 `\n` 继续切
3. 如果切完还太大，再在这个 unit 内按中文句号 `。` 继续切
4. 最终得到一批大小不一的 unit
5. 后面再把多个 unit 合并成最终 chunk

例如一段 1000 字：

1. 先看能不能按空行切
2. 不行就看能不能按换行切
3. 还不行就按句号切

所以：

```text
separators 决定怎么拆 unit
chunk_size 决定 unit 合并成 chunk 时的目标大小
chunk_overlap 决定相邻 chunk 保留多少重叠内容
```

---

## 10. 为什么说要“保留自然边界”

意思是：

**能在段落结束处切，就不要在句子中间切。**

比如：

```text
RAG 会先检索相关内容，再让大模型基于这些内容回答问题。
```

如果硬切，可能变成：

```text
chunk 1：RAG 会先检索相关内容，再让大模型
chunk 2：基于这些内容回答问题。
```

这样语义被割裂了。

项目更希望：

1. 先在段落边界切
2. 再在换行边界切
3. 再在句号边界切

这样每个 chunk 更像一个完整语义单元。

---

## 11. 表格、图片、代码块为什么要特殊保护

有些内容不能随便从中间切开，例如：

1. Markdown 表格
2. 图片引用
3. 代码块
4. 公式块
5. Markdown 链接

如果从中间切开：

1. 表格结构会坏
2. 图片语法会坏
3. 代码块上下文会坏
4. 公式会变残缺

所以项目在切 unit 前，会先把这类内容识别出来，尽量作为整体保留。

---

## 12. 如果一个 unit 太大怎么办

这里分几种情况。

### 1. 还有更细分隔符

继续递归切。

例如：

1. 空行切完还有 1000 字
2. 再按换行切
3. 还大，再按句号切

### 2. 所有分隔符都用完了，还是太大

那这个 unit 会暂时保留下来。

项目不会为了严格满足 `512`，马上在中间硬砍。

所以最后有可能出现：

```text
chunk_size = 512
但某个 chunk 实际有 700 或 1000 字
```

因为项目更重视语义边界，而不是死守固定长度。

### 3. 特别极端地大

如果单个 unit 超过绝对上限 `7500` 字符，源码会强制拆开：

1. 先以 7500 为切点
2. 再往前最多找 200 个字符
3. 优先找换行或空格
4. 找到就在自然边界处断开
5. 找不到就在 7500 处硬切

这个是防止极端超长内容撑爆后续 embedding 的保险机制。

---

## 13. overlap 是怎么起作用的

当一个 chunk 输出后，新 chunk 不会完全从零开始。

它会尽量保留上一块末尾的一些 unit，作为 overlap。

默认：

```text
80 个字符
```

作用：

1. 保留跨边界语义
2. 避免一句话前半段和后半段完全分家
3. 提高召回时命中完整语义的概率

但是它不会无脑复制 80 个字符。

源码会基于 unit 来保留 overlap，并且会跳过纯空白、纯分隔符这类没有意义的内容。

---

## 14. chunk 最终保存什么

最终生成的 chunk 会保存：

1. `Content`
   chunk 原文内容

2. `Seq`
   chunk 顺序

3. `Start / End`
   在原文中的位置

4. `ContextHeader`
   标题上下文，主要在 heading 策略里使用

例如：

```md
# RAG
## 文档入库
这里是正文
```

正文 chunk 在 embedding 时，可以带上：

```text
# RAG
## 文档入库
```

这样即使正文很短，模型也知道它属于哪个章节。

但原始 `Content` 仍然尽量保持真实文本，不随便污染原文。

---

## 15. 图片在文档切分里怎么处理

如果 md 中有图片，例如：

```md
# 标题 1

这里是一段正文。

![架构图](local://xxx.png)
```

### 在切分阶段

`heading` 只会先得到一个普通文本 chunk：

```text
标题 1 + 正文 + 图片引用
```

图片引用先跟着正文保留。

### 如果开启多模态处理

后续异步任务会对图片做：

1. OCR
2. Caption

然后额外生成：

1. `image_ocr` chunk
2. `image_caption` chunk

这两个图片 chunk 会通过 `parent_chunk_id` 挂到原来的 text chunk 下面。

注意：

这是一种**图片信息挂载关系**，不是普通意义上的“父子分块策略”。

---

## 16. 父子分块策略是什么

项目还支持真正的父子分块：

1. parent chunk
   大块，用于返回更完整上下文

2. child chunk
   小块，用于 embedding 和精确检索

查询时：

1. 用 child chunk 做召回
2. 命中后可以返回 parent chunk 的更大上下文

它解决的是：

```text
小块更容易检索准
大块更适合给 LLM 阅读
```

---

## 17. 切分完成后怎么进入 RAG

切分结束后：

1. chunk 原文保存到业务数据库的 `chunks` 表
2. chunk 文本送给 embedding 模型生成向量
3. 向量和 `chunk_id` 写入检索引擎
4. 用户提问时，用 query embedding 做相似度检索
5. 拿到相关 `chunk_id`
6. 再回数据库查 chunk 原文
7. 把这些 chunk 作为上下文交给大模型生成答案

---

## 18. 面试时最推荐的总结方式

你可以这样讲：

1. 项目先把不同格式文档统一解析成 `MarkdownContent`
2. 切分时不是固定长度硬切，而是优先保留文档结构和自然边界
3. 支持 `auto`、`heading`、`heuristic`、`legacy/recursive` 几种策略
4. 普通递归切分会先按分隔符拆成 `unit`，再把多个 unit 合并成 chunk
5. `chunk_size` 控制目标大小，`overlap` 保留边界上下文
6. 如果结构化策略效果不好，会回退到更稳定的递归切分
7. 最终 chunk 原文存数据库，向量写入检索引擎，作为 RAG 召回的基本单位
