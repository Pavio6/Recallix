# 项目中的 System Prompt

本文只统计当前项目里**真正以 `role = "system"` 发送给模型**的内容，并说明它们的代码位置、使用场景和设计原因。

## 1. 普通 RAG 回答 Prompt

- **代码位置**：`internal/chat/prompt/prompt.go` 中的 `BuildSystemPrompt()`
- **调用位置**：`internal/chat/service/chat.go`
- **使用场景**：当前问题需要进入知识库检索链路时，作为最终回答模型的主 system prompt。

### Prompt 核心内容

1. 你是 `Recallix`，可以访问知识库。
2. 有知识库上下文时，优先根据上下文回答。
3. 不知道时要明确说不知道，不要编造。
4. 输出要使用正确的 Markdown；表格要有合法分隔行。
5. 回答要简洁，但不能过于敷衍。
6. 如果提供了长期记忆，要结合用户长期记忆作答。

### 为什么这样写

- **先定义角色**：让模型知道自己不是普通聊天机器人，而是“带知识库能力的助手”。
- **要求优先依据上下文**：这是 RAG 的核心，否则模型可能忽略检索结果，直接依赖自身参数知识。
- **要求不知道就承认不知道**：降低幻觉风险，尤其是在知识库没有命中时。
- **约束 Markdown**：这是面向前端渲染的工程性要求，避免模型把表格输出坏。
- **“简洁但完整”**：平衡两种常见问题：过度啰嗦和过度简略。
- **提醒使用长期记忆**：让个性化信息不是“附带塞进 prompt 却被忽略”，而是成为显式行为约束。

## 2. 非检索场景 Prompt

- **代码位置**：`internal/chat/prompt/prompt.go` 中的 `BuildNoRetrievalSystemPrompt(intent string)`
- **调用位置**：`internal/chat/service/chat.go`
- **使用场景**：查询理解阶段判断当前轮不需要检索知识库时，替代普通 RAG prompt。

### 2.1 `greeting`

- **适用意图**：问候、感谢、告别。
- **核心要求**：自然、简短回复；除非用户主动问，否则不要提知识库或检索。

**为什么这样写**：
- 问候类问题如果还把知识库搬出来，会显得很机械。
- “简短”能避免模型对一句“你好”输出过长内容。

### 2.2 `chitchat`

- **适用意图**：普通闲聊。
- **核心要求**：自然聊天；不主动提知识库或检索。

**为什么这样写**：
- 闲聊的目标是自然交互，不是证明系统有 RAG 能力。
- 防止模型在无关问题上说“根据知识库……”，造成错觉。

### 2.3 `follow_up`

- **适用意图**：只靠已有对话历史就能回答的追问。
- **核心要求**：基于现有历史回答；没有检索上下文时，不要声称查过知识库。

**为什么这样写**：
- 追问通常依赖上一轮回答，不一定需要重新检索。
- 明确禁止“伪检索声明”，能避免模型把历史推理包装成知识库检索结果。

### 2.4 `summarize`

- **适用意图**：总结当前对话。
- **核心要求**：只总结当前聊天历史；不要声称查过知识库。

**为什么这样写**：
- 用户要的是“总结我们刚才聊了什么”，不是“总结知识库”。
- 这能避免任务目标漂移。

### 2.5 默认兜底

- **适用场景**：非上述几类，但当前轮仍决定不走检索。
- **核心要求**：自然、诚实回答；没有检索上下文时，不要声称查过知识库。

**为什么这样写**：
- 给未知意图留一个安全出口。
- 保持一个关键原则：**没有检索，就不要假装检索过**。

## 3. 查询理解 Prompt

- **代码位置**：`internal/chat/query/rewrite.go` 中的 `queryUnderstandingPrompt`
- **调用位置**：`internal/chat/query/rewrite.go` 中的 `Understand(...)`
- **使用场景**：在正式回答前，先把用户问题改写成自包含问题，并判断是否需要检索。

### Prompt 核心内容

1. 只做两件事：
   - 改写最新问题为简洁、自包含的问题。
   - 判断用户意图。
2. 只能从固定意图集中选择：
   - `greeting`
   - `kb_search`
   - `clarification`
   - `follow_up`
   - `summarize`
   - `chitchat`
3. 默认不确定时走 `kb_search`。
4. 有礼貌词不代表就是 `greeting`。
5. `follow_up` 只能在纯靠历史就能回答时使用。
6. `rewrite_query` 要保留实体和核心检索词。
7. 强制只返回 JSON。
8. 提供多个中英文混合示例。

### 为什么这样写

- **把“改写”和“路由”合并**：少一次模型调用，也让改写结果和意图判断共享上下文。
- **固定意图集合**：避免模型自由发挥，方便后端做确定性分支。
- **默认 `kb_search`**：这是保守策略，宁可多查一次，也不希望把本应检索的问题误判成闲聊。
- **专门提醒 polite words**：例如“你好，请告诉我线程和进程的区别”，核心仍是事实查询，不该被“你好”误导。
- **严格限制 `follow_up`**：否则系统会因为误判而跳过本应发生的检索。
- **要求保留实体和检索词**：查询改写的目标不是“语言优美”，而是提升召回效果。
- **JSON only**：后端需要稳定解析，减少自由文本带来的脆弱性。
- **给示例**：few-shot 示例能显著降低模型在边界场景下的漂移。

## 4. 候选记忆提取 Prompt

- **代码位置**：`internal/memory/memory.go` 中的 `ExtractCandidate(...)`
- **调用位置**：`internal/memory/memory.go` 中的 `ProcessExtraction(...)`
- **使用场景**：一轮问答结束后，从当前对话中判断是否存在值得长期保存的用户记忆。

### Prompt 核心内容

1. 判断当前对话中是否存在“稳定、长期值得记住”的用户信息。
2. 只返回 JSON：
   - `action = create | skip`
   - `memory_type = preference | profile | project | fact`
   - `memory_text`
3. 如果没有值得记住的内容，返回 `skip`。
4. 如果有，尽量抽成第一人称、简洁事实。
5. 明确四类 memory 的含义。

### 为什么这样写

- **强调“稳定、长期”**：防止把临时情绪、一次性要求、短期噪声都写入长期记忆。
- **只做 `create / skip`**：第一阶段先判断“有没有候选记忆”，把复杂决策留给下一步。
- **结构化输出**：方便后端稳定解析，也让后续治理更容易。
- **引入 memory type**：让长期记忆不再全是扁平 `fact`，为后续差异化召回和管理做准备。
- **第一人称**：记忆注入 prompt 时更自然，也更贴合“这是用户自己的长期信息”。

## 5. 记忆决策 Prompt

- **代码位置**：`internal/memory/memory.go` 中的 `Decide(...)`
- **调用位置**：`internal/memory/memory.go` 中的 `ProcessExtraction(...)`
- **使用场景**：当系统已经找到了相似旧记忆时，判断新候选记忆应该新增、更新还是跳过。

### Prompt 核心内容

1. 给模型一个候选记忆和若干相似旧记忆。
2. 只能选择：
   - `create`
   - `update`
   - `skip`
3. 只返回 JSON：
   - `action`
   - `target_memory_id`
   - `memory_type`
   - `memory_text`
4. 如果是 `update`，必须指定现有记忆 ID。
5. 如果是 `create`，`target_memory_id` 必须为空。
6. 近重复优先 `skip`。
7. 如果新记忆替代或实质性修正旧记忆，优先 `update`。

### 为什么这样写

- **把“是否值得记住”和“如何治理已有记忆”拆开**：能降低单次 prompt 复杂度，让决策更稳定。
- **强制绑定 `target_memory_id`**：保证 update 是可执行动作，而不是一句模糊判断。
- **近重复优先 skip**：防止长期记忆无限膨胀。
- **支持 update**：让系统能处理用户偏好变化、项目状态变化，而不是只会追加新事实。

## 6. 动态 System Context：用户记忆上下文

- **代码位置**：`internal/chat/prompt/prompt.go` 中的 `BuildMemoryContext(...)`
- **调用位置**：`internal/chat/service/chat.go`
- **使用场景**：当前问题召回到相关长期记忆时，作为一条 system message 注入最终回答模型。

### 实际格式

```text
【User Memory】
- 记忆 1
- 记忆 2
```

### 为什么这样写

- **单独作为 system message**：告诉模型这些是高优先级背景信息，而不是普通用户发言。
- **用明确标题分区**：让模型知道这部分不是知识库事实，而是用户长期信息。
- **列表格式**：对多条短记忆最清晰，模型也容易逐条利用。

---

## 7. 动态 System Context：知识库上下文

- **代码位置**：`internal/chat/prompt/prompt.go` 中的 `BuildContext(...)`
- **调用位置**：`internal/chat/service/chat.go`
- **使用场景**：RAG 检索命中后，把最终保留的 chunk 作为 system message 注入回答模型。

### 实际格式

```text
【Knowledge Base Context】
[Source 1] # 标题路径
正文内容
[Source 2] 正文内容
```

### 为什么这样写

- **单独标识为知识库上下文**：提醒模型这些是本轮回答最应依赖的外部事实。
- **保留 `Source N`**：后续如果需要正文引用或前端对齐来源，已经有了稳定编号。
- **保留标题路径**：让模型知道 chunk 属于哪一章哪一节，减少脱离上下文的误解。

---

## 8. 当前 System Message 的组装顺序

- **代码位置**：`internal/chat/service/chat.go`

最终回答阶段当前按以下顺序组装消息：

1. 主 system prompt
2. `User Memory` system context（如果有）
3. `Knowledge Base Context` system context（如果有）
4. 最近对话历史

### 为什么这样排

- **先放规则，再放资料**：先告诉模型“怎么答”，再给它“依据什么答”。
- **先放用户记忆，再放知识库内容**：当前实现把个性化背景放在事实证据之前，让回答先符合用户，再结合知识库事实。
- **最后放对话历史**：让模型在回答时仍能保持上下文连续。

## 小结

当前项目中的 system prompt 可以分成三类：

1. **回答控制类**：普通 RAG prompt、no-retrieval prompt。
2. **任务控制类**：查询理解 prompt、记忆提取 prompt、记忆决策 prompt。
3. **动态上下文类**：用户记忆上下文、知识库上下文。

学习时可以重点观察三件事：

- 它要解决的**具体问题**是什么。
- 它把模型的输出空间**限制到了什么程度**。
- 它有没有把“模型说什么”和“后端如何稳定消费结果”一起考虑进去。
