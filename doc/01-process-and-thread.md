# 进程与线程：操作系统中的执行单位

## 进程的资源边界

进程是操作系统进行资源分配和保护的基本单位。一个正在运行的程序通常会被操作系统表示为一个进程，进程拥有独立的虚拟地址空间、打开的文件描述符、环境变量、权限信息以及调度状态。用户启动一个应用时，系统会为它创建进程控制块，记录进程标识符、寄存器上下文、内存映射、优先级、父子关系和统计信息。进程的独立性让不同程序之间不容易互相破坏数据，也让操作系统可以在程序崩溃时只回收该进程的资源，而不影响整个系统。

## 线程的共享模型

线程是进程内部的执行流。一个进程可以只有一个线程，也可以包含多个线程。线程之间共享进程的地址空间、全局变量、堆、打开文件和部分内核对象，但每个线程都有自己的栈、寄存器状态和线程局部数据。线程的引入主要是为了提高并发能力和降低上下文切换成本。例如浏览器可以用多个线程分别负责界面刷新、网络请求、脚本执行和磁盘缓存；服务器程序可以用线程池同时处理大量客户端连接。

## 调度状态与阻塞

进程和线程都需要调度。调度器会根据就绪队列、优先级、时间片、阻塞状态和 CPU 亲和性等因素决定下一个运行实体。进程或线程运行时，如果主动等待 I/O、锁、信号量、条件变量或定时器，就会进入阻塞状态；当等待条件满足后，内核再把它放回就绪队列。时间片耗尽时，内核会触发抢占，把当前执行上下文保存起来，再恢复另一个线程或进程的上下文。

## 上下文切换成本

上下文切换是理解并发性能的关键。线程切换通常比进程切换轻一些，因为同一进程内的线程共享地址空间，切换时不一定需要更换页表。进程切换往往涉及地址空间变化，可能导致 TLB 失效，从而增加内存访问开销。不过线程共享内存也带来了数据竞争问题。多个线程同时读写同一变量，如果没有互斥锁、原子操作或其他同步机制，就可能出现不可预测结果。

## 进程间通信方式

进程间通信通常需要操作系统介入。常见方式包括管道、消息队列、共享内存、套接字、信号、文件和内存映射。共享内存速度快，但需要额外同步；套接字适合本机和网络通信；管道适合父子进程或命令行流水线。线程间通信因为共享地址空间更直接，但也更容易出错，需要谨慎设计临界区。

## 现代系统中的任务模型

现代操作系统通常把线程作为调度单位，把进程作为资源容器。Linux 中的进程和线程底层都可以看作 task，差异主要体现在资源共享标志上。Windows 也区分进程对象和线程对象，线程被调度器实际运行，进程负责承载地址空间和句柄表。理解这个模型有助于分析程序卡顿、CPU 占用高、死锁和并发吞吐问题。

## 并发模型选择

在实际开发中，应该根据任务性质选择并发模型。CPU 密集型任务通常不宜创建过多线程，线程数量接近 CPU 核数更容易获得稳定性能。I/O 密集型任务可以使用更多线程，也可以使用事件循环、异步 I/O 或协程来减少阻塞成本。无论选择哪种方式，都要关注共享状态、取消机制、资源释放和错误传播。进程与线程不是越多越好，合理的结构比盲目并发更重要。
